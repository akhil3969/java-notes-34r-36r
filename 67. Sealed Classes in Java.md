

# 🔒 Sealed Classes in Java

---

### 1. Definition

A **Sealed Class** in Java is a class or interface that **restricts which other classes or interfaces can extend or implement it**.

It was introduced in **Java 15 (Preview)** and became **official in Java 17**.

> *Sealed classes give the developer control over inheritance.*

---

### 2. Why Sealed Classes?

Before sealed classes:

* Any class could extend a parent class
* Hard to control class hierarchies
* Risky for frameworks and domain models

With sealed classes:

* Only **known, fixed subclasses** are allowed
* Safer and more predictable design
* Better support for **pattern matching**

---

### 3. Declaration Syntax

```java
sealed class Parent permits Child1, Child2, Child3 {
}
```

Subclasses **must** be declared as:

* `final`
* `sealed`
* `non-sealed`

---

### 4. Simple Example

```java
sealed class Vehicle permits Car, Bike {
}

final class Car extends Vehicle {
}

final class Bike extends Vehicle {
}
```

✔ Only `Car` and `Bike` can extend `Vehicle`
❌ Any other class cannot extend it

---

### 5. Rules for Sealed Classes

✔ A sealed class **must declare `permits` clause**
✔ All permitted subclasses must be **in the same module or package**
✔ Every permitted subclass must be:

* `final` OR
* `sealed` OR
* `non-sealed`

---

### 6. `non-sealed` Keyword

`non-sealed` **removes restriction** and allows further inheritance.

```java
sealed class Shape permits Circle, Rectangle {
}

non-sealed class Circle extends Shape {
}

final class Rectangle extends Shape {
}
```

Now:

```java
class ColoredCircle extends Circle { } // ✅ Allowed
```

---

### 7. Sealed Interface Example

```java
sealed interface Payment permits CreditCard, UPI {
}

final class CreditCard implements Payment {
}

final class UPI implements Payment {
}
```

✔ Only listed implementations are allowed

---

### 8. Comparison: Sealed vs Non-Sealed vs Final

| Feature             | Sealed Class           | Non-Sealed Class     | Final Class        |
| ------------------- | ---------------------- | -------------------- | ------------------ |
| Inheritance Control | Restricted             | Open                 | Completely blocked |
| Can be extended     | Only permitted classes | Any class            | ❌ No               |
| Used for            | Controlled hierarchies | Flexible hierarchies | Immutable design   |
| Java Keyword        | `sealed` + `permits`   | `non-sealed`         | `final`            |

---

### 9. Internal Working (Conceptual)

```java
sealed class Animal permits Dog, Cat { }
```

The compiler ensures:

* Only `Dog` and `Cat` extend `Animal`
* Any other extension causes **compile-time error**

```java
class Lion extends Animal { } // ❌ Compile-time error
```

---

### 10. Sealed Classes with Pattern Matching (Java 17+)

```java
sealed class Shape permits Circle, Square {}

final class Circle extends Shape {
    double radius;
}

final class Square extends Shape {
    double side;
}
```

```java
static double area(Shape shape) {
    return switch (shape) {
        case Circle c -> Math.PI * c.radius * c.radius;
        case Square s -> s.side * s.side;
    };
}
```

✔ Compiler knows **all possible subclasses**
✔ No `default` case required

### 11. Real-World Use Cases

| Scenario                   | Example                       |
| -------------------------- | ----------------------------- |
| Payment Systems            | Card, UPI, Wallet types       |
| Domain-Driven Design (DDD) | Fixed business rules          |
| Framework Design           | Controlled extension points   |
| Expression Trees           | AST nodes, compiler internals |
| State Machines             | Fixed states and transitions  |

---

### 12. Limitations of Sealed Classes

❌ Cannot be anonymous classes
❌ More rigid than normal inheritance
❌ All subclasses must be known at compile time
❌ Slightly more verbose than normal classes

---

### 13. Sealed vs Abstract Class

| Feature       | Sealed Class                   | Abstract Class     |
| ------------- | ------------------------------ | ------------------ |
| Inheritance   | Restricted                     | Open               |
| Instantiation | Cannot instantiate             | Cannot instantiate |
| Control       | Strong control over subclasses | No control         |
| Use Case      | Fixed hierarchy                | Base functionality |
